这一页聚焦三件事：怎样测，怎样攻防对照，怎样保证交付链可信。对于带工具调用、代码执行、外部检索与自动文稿生成的科研系统，这三条线通常需要一起建设。

## 六、适合科研系统的最小评测闭环

1. 用 `promptfoo` 或 `garak` 建立基础安全回归集。
2. 用 `inspect_ai` 或 `OpenAI Evals` 把安全测试写成可重复任务。
3. 对工具调用和 `MCP` 场景加入 `evil-mcp-server`、注入样本和越权样本。
4. 对 notebook、脚本、容器和模型文件加入 `NB Defense`、`modelscan`、`Trivy` 或 `Grype`。
5. 对正式交付物补 `SLSA`、`Sigstore`、`SPDX / CycloneDX` 与 provenance 记录。

## 七、推荐组合

1. `garak + promptfoo + inspect_ai`
   适合快速建立内容安全、注入攻击和任务回归闭环。
2. `PyRIT + HarmBench + JailbreakBench`
   适合围绕有害输出、越狱和更系统的红队测试做对照。
3. `evil-mcp-server + MCP Security Best Practices + MCP Authorization`
   适合围绕 `MCP` 宿主和远程工具调用做专项加固。
4. `modelscan + NB Defense + Trivy / Grype`
   适合检查模型文件、notebook、镜像和依赖方向的供应链风险。
5. `SLSA + Sigstore + SPDX / CycloneDX + in-toto`
   适合把研究交付链做成可验证、可追溯状态。
